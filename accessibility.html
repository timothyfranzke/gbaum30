Text Legibility & Layout Integrity at High Zoom Levels
AA AAA
Purpose
The primary purpose of this manual accessibility test is to meticulously evaluate whether a digital platform's content, including text and layout, remains fully legible, functional, and maintains its integrity when users apply increased zoom levels, text spacing adjustments, or custom visual presentation settings. This test is designed to be performed after an automated accessibility scan and aims to uncover complex visual and functional issues that arise from content reflow, element overlap, insufficient target sizes at magnification, and conflicts with user-defined color preferences. Ensuring content adapts gracefully to user-controlled visual settings is crucial for individuals with low vision, cognitive disabilities, or specific reading preferences, preventing loss of information, unusable interfaces, and a frustrating user experience..
WCAG Criteria Covered

This test specifically addresses the following WCAG 2.2 Success Criteria related to text legibility, layout integrity, and zoom:

  • 1.4.4 Resize text AA

    This criterion states that text can be resized without assistive technology up to 200 percent without loss of content or functionality. This means that users should be able to increase the font size (via browser settings or dedicated controls) to 200% of its original size without content or functionality becoming inaccessible.

    • Relevance/Testing:

      We will use browser zoom functionality to increase the text size (or full page zoom) to 200% and carefully observe all content elements (text, images, interactive components) to ensure no content is cut off, overlaps, or disappears, and all functionality remains intact.

  • 1.4.8 Visual Presentation AAA

    This criterion requires that for the visual presentation of blocks of text, a mechanism is available to achieve the visual presentation in a way that foreground and background colors can be selected by the user. This applies to cases where users override default colors (e.g., high contrast modes).

    • Relevance/Testing:

      We will test the platform's content by applying user-selected foreground and background colors (e.g., via browser settings, operating system high contrast mode, or custom stylesheets). We will verify that all text remains visible, legible, and maintains sufficient contrast against the new background, and that no content or functionality is lost or obscured due to these color changes. This includes checking text, icons, and interactive elements.

  • 1.4.10 Reflow AA

    This criterion requires that content can be presented without loss of information or functionality, and without requiring scrolling in two dimensions (horizontal scrolling), for vertical scrolling content at a width equivalent to 320 CSS pixels. This typically corresponds to a 400% zoom level on a standard desktop browser (1280px effective width becoming 320px). For mobile, it means the content reflows vertically without horizontal scrolling in portrait mode.

    • Relevance/Testing:

      We will push browser zoom levels up to 400% (or simulate a narrow viewport width of 320px on desktop). We will then verify that all content reflows into a single column (or a layout that only requires vertical scrolling), and that no horizontal scrollbar appears. We will check for content overlap, truncation, or disappearance, and ensure all interactive elements remain usable.

  • 1.4.12 Text Spacing AAA

    This criterion states that in content implemented using markup languages that support the following text style properties, no loss of content or functionality occurs by setting all of the following: line height (line spacing) to at least 1.5 times the font size; paragraph spacing to at least 2 times the font size; letter spacing (tracking) to at least 0.12 times the font size; and word spacing to at least 0.16 times the font size. This ensures users can override default text spacing for improved readability.

    • Relevance/Testing:

      We will use browser extensions or custom CSS to apply these specific text spacing overrides. We will then examine all textual content to ensure that increasing line height, paragraph spacing, letter spacing, and word spacing does not cause text to overlap, become cut off, or break the layout in a way that leads to loss of content or functionality.

  • 2.5.8 Target Size (Minimum) AAA

    This criterion requires that the size of the target for pointer inputs is at least 24px by 24px CSS pixels, except for certain exceptions (e.g., if an equivalent link is available in an adequate size, if it's text-only, or if it's constrained by the browser). This ensures interactive elements are large enough for users with motor impairments or those using non-precision pointers (e.g., touchscreens).

    • Relevance/Testing:

      While testing at high zoom levels, interactive elements (buttons, links, form fields) will naturally increase in size. We will visually verify that all interactive targets meet or exceed the 24px×24px minimum when viewed at standard scale, ensuring they remain adequately sized even if browser zoom scaling isn't perfectly proportional. For mobile, we will verify this with touch.

Test Environment

This test primarily focuses on browser-level zoom and responsiveness, requiring both desktop and mobile/tablet environments..

  • Desktop:
    • Operating Systems: Windows (latest two versions), macOS (latest two versions).
    • Browsers: Latest stable versions of Chrome, Firefox, Edge, Safari (with built-in zoom functionality and developer tools).
    • Input Methods: Standard mouse (for visual inspection), keyboard (for navigation).
    • Tools: Browser extensions for applying custom CSS (e.g., "Stylebot," "User CSS") or specific text spacing overrides, if not directly available through browser settings.
  • Mobile & Tablet:
    • Operating Systems: iOS (latest two versions), Android (latest two versions).
    • Browsers: Default browsers (Safari on iOS, Chrome on Android) with pinch-to-zoom capabilities.
    • Input Methods: Touchscreen (for pinch-to-zoom, interaction with scaled elements).
    • Developer Tools: Remote debugging capabilities (e.g., Chrome DevTools for Android, Safari Web Inspector for iOS) for simulating viewport widths or inspecting element sizes.
    • Orientation: Portrait and Landscape modes (to assess reflow behavior).
    • Settings: Device-level accessibility settings for high contrast or custom colors.
Test Preparation

Before commencing the manual test, ensure the following steps are completed:

  1. Identify All Content Areas:

    Systematically list all pages, components, and specific interactive elements on the platform that contain text or interactive elements. This includes:

    • Standard body text, headings, paragraphs.
    • Navigation menus, links, buttons.
    • Form fields, labels, instructions, error messages.
    • Tables, data grids.
    • Modals, pop-ups, alerts.
    • Images with overlaid text or captions.
    • Any areas with complex layouts or dense information.

  2. Understand Responsive Design:

    Familiarize yourself with how the platform is expected to adapt to smaller screen sizes or increased zoom levels.

  3. Establish Baseline Functionality:

    Briefly interact with various content areas at default zoom to understand their normal appearance and behavior.

  4. Familiarize with WCAG Criteria:

    Re-read and internalize the specific requirements of WCAG 1.4.4, 1.4.8, 1.4.10, 1.4.12, and 2.5.8. Understand the numerical values for text spacing (1.5, 2, 0.12, 0.16).

  5. Prepare for Browser Zoom/Text Resizing:

    Know how to use your chosen browsers' zoom features (Ctrl/Cmd + Plus/Minus, or browser settings).

  6. Prepare for Text Spacing Override:

    If using a browser extension for text spacing, have it ready and know how to apply the specified overrides.

  7. Prepare for Custom Color Testing:

    Know how to activate operating system high contrast modes or use browser extensions to apply custom foreground and background colors.

  8. Clear Browser Cache/Cookies:

    To ensure a clean test environment, clear browser cache and cookies as needed.

Step-by-Step Testing Process

Perform the following steps for each identified form on both desktop and mobile/tablet environments (with an external keyboard where applicable). Document observations meticulously.

STEP 1 General Procedure for Zoom Functionality (WCAG 1.4.4, 1.4.10)
Test 200% Zoom (WCAG 1.4.4):
  • Load the page at 100% zoom.
  • Increase browser zoom to 200%.
  • Test Case:
    Does all content (text, images, interactive elements) remain fully visible and legible?
  • Test Case:
    Does the layout adapt without horizontal scrolling?
  • Test Case:
    Is all functionality still present and operable? Can all links, buttons, and form fields be used?
  • Test Case:
    Do elements overlap or become truncated?
  • Test Case:
    Does text reflow gracefully within its containers without breaking words or characters?
  • Test Case (Sticky Elements):
    If there are sticky headers, footers, or sidebars, do they obscure content or focused elements when zoomed?
Test 400% Zoom / Reflow (WCAG 1.4.10):
  • Load the page at 100% zoom.
  • Increase browser zoom to 400% (or simulate a 320px effective viewport width if more precise).
  • Test Case:
    Does the content reflow into a single column (or a layout requiring only vertical scrolling)? Is there nohorizontal scrollbar?
  • Test Case:
    Is all information still present? Is anything hidden, truncated, or removed?
  • Test Case:
    Is all functionality still present and operable? Can all interactive elements be used without horizontal scrolling?
  • Test Case:
    Are there any instances of overlapping content or distorted layouts?
  • Test Case (Image Scaling):
    Do images scale appropriately without breaking the layout or becoming excessively large/small?
Mobile/Tablet Specific Zoom (Pinch-to-Zoom):
  • On a mobile or tablet device, load the page.
  • Use pinch-to-zoom to zoom in significantly (e.g., to match 200 or 400 equivalent magnification).
  • Test Case:
    Does pinch-to-zoom function smoothly and consistently across the page?
  • Test Case:
    Does content adapt responsively, reflowing as appropriate for the zoomed level, similar to desktop browser zoom?
  • Test Case:
    Are interactive elements still tappable, and is the overall functionality maintained?
STEP 2 General Procedure for Visual Presentation (WCAG 1.4.8 AAA)
Apply User-Selected Colors:
  • Activate an operating system's high contrast mode (e.g., Windows High Contrast Mode, macOS Invert Colors/Smart Invert).
  • Alternatively, use a browser extension to apply custom foreground and background colors (e.g., light text on a dark background, or vice-versa) to the entire page.
Observe Text and Layout Integrity:
  • Carefully examine all content on the page, including text, images, icons, and interactive elements, under these custom color settings.
  • Test Case:
    Does all text remain visible and legible? Does it maintain sufficient contrast against the new background colors?
  • Test Case:
    Do icons and graphical elements (e.g., charts, progress bars) remain visible and convey their meaning? Do they adapt their colors or outlines to ensure contrast?
  • Test Case:
    Is any content (text, images, or parts of the layout) hidden or rendered invisible due to color conflicts with the user's chosen theme (e.g., text set to color: white; on a background-color: white; in a high-contrast mode)?
  • Test Case:
    Does all functionality remain present and operable? Can all links, buttons, and form fields be used, and are their states (e.g., hover, focus, selected) still perceivable?
  • Test Case:
    Are there any instances where background-image or border-image properties prevent text from being visible or legible in custom color modes?
STEP 3 General Procedure for Text Spacing (WCAG 1.4.12)
Apply Text Spacing Overrides:
Use a browser extension or custom CSS to apply the following styles to all text on the page:
  • line-height: 1.5;
  • word-spacing: 0.16em;
  • letter-spacing: 0.12em;
  • margin-bottom (for paragraphs and block elements, simulating 2em paragraph spacing).

Note: Ensure the override is applied globally and consistently.

Observe Text and Layout Integrity:
  • Carefully examine all textual content, including headings, paragraphs, list items, labels, and text within buttons/links.
  • Test Case:
    Does any text overlap other text, images, or interactive elements?
  • Test Case:
    Does any text become truncated or cut off (e.g., text overflowing its container)?
  • Test Case:
    Does the increased text spacing cause any interactive elements (buttons, links) to break, overlap, or become unusable?
  • Test Case:
    Does the overall page layout remain functional and usable, or does it break significantly due to increased spacing?
  • Test Case:
    Are there any !important declarations or fixed pixel widths/heights on text containers that prevent text spacing from applying correctly?
STEP 4 General Procedure for Target Size (WCAG 2.5.8)
Inspect Interactive Elements at Default Zoom:
  • At 100% zoom on desktop or standard mobile view, visually scan all interactive elements (buttons, links, form fields, icons used as buttons).
  • Test Case:
    Estimate the size of each interactive target. Does it appear to be at least 24px by 24px? (For precise measurements, use developer tools to inspect computed styles).
  • Test Case (Mobile/Tablet):
    On touch devices, verify that touch targets are large enough and have sufficient spacing to be easily tapped without accidentally activating adjacent elements.
  • Note:
    Text links are often excluded if they are purely text, but if they are graphical elements or icons, this applies.
Re-evaluate Target Size at Zoomed Levels:
  • Revisit problematic areas at 200% and 400% zoom.
  • Test Case:
    Do the increased zoom levels ensure that targets become even larger and more easily clickable, or do any scaling issues inadvertently reduce effective target size?
Identifying Issues

During the testing process, identify and document any deviations from the expected behavior based on WCAG 1.4.4, 1.4.8, 1.4.10, 1.4.12, and 2.5.8. Examples of common issues include:

  • Content Loss/Truncation on Zoom (WCAG 1.4.4, 1.4.10 Failure):
    Text or images get cut off, hidden, or overlap other content when zoomed to 200% or 400%.
  • Horizontal Scrolling on Zoom (WCAG 1.4.10 Failure):
    A horizontal scrollbar appears at 400% zoom (or 320px equivalent width), requiring users to scroll both horizontally and vertically.
  • Broken Layout on Zoom (WCAG 1.4.4, 1.4.10 Failure):
    Elements overlap, become distorted, or render incorrectly, making the page unusable or unreadable at high zoom levels.
  • Loss of Functionality on Zoom (WCAG 1.4.4, 1.4.10 Failure):
    Interactive elements (buttons, links, form fields) become unclickable, disappear, or are otherwise non-functional when zoomed.
  • Content Invisible/Illegible with Custom Colors (WCAG 1.4.8 AAA Failure)
    Text, icons, or other essential content becomes invisible, loses contrast, or is unreadable when user-selected foreground/background colors are applied (e.g., in high contrast mode).
  • Functionality Lost with Custom Colors (WCAG 1.4.8 AAA Failure):
    Interactive elements become unusable or their states are not perceivable when user-selected colors are applied.
  • Text Overlap/Truncation with Text Spacing (WCAG 1.4.12 Failure):
    Applying text spacing overrides causes text to overlap, get cut off, or break out of its containers.
  • Layout Breakage with Text Spacing (WCAG 1.4.12 Failure):
    The overall page layout or specific components (e.g., buttons with text, navigation items) break when text spacing is increased.
  • Fixed Dimensions Preventing Reflow (WCAG 1.4.10, 1.4.12 related):
    The use of fixed pixel widths/heights on containers or elements prevents content from reflowing or text spacing from taking effect.
  • Small Interactive Targets (WCAG 2.5.8 Failure):
    Interactive elements are smaller than 24px×24px CSS pixels, making them difficult to activate, especially on touch devices or for users with motor impairments.
  • Sticky Elements Obscuring Content (WCAG 2.4.11 - related):
    Fixed headers, footers, or sidebars obscure content or focused elements when zoomed.
  • Images of Text:
    While covered in another test, images of text will pixelate or become illegible when zoomed, exacerbating issues (WCAG 1.4.5 related).
Severity Classification

Assign a severity level to each identified issue based on its impact on user accessibility:

  • Critical:
    Leads to complete loss of essential content or functionality when zoomed to 200% or 400%. Requires horizontal scrolling at 400% zoom, or makes core tasks impossible.
  • High:
    Significantly hinders users from accessing or using content effectively at high zoom levels or with text spacing overrides. Causes major layout breakage, extensive overlap, or makes key functionality very difficult to operate.
  • Medium:
    Causes moderate inconvenience or frustration. Some minor content overlap or truncation occurs, or text spacing causes minor layout issues but doesn't completely break functionality.
  • Low:
    Minor cosmetic issues or slight inefficiencies. Minimal impact on usability or readability at high zoom levels or with text spacing.
Final Note

Ensuring text legibility and layout integrity at high zoom levels, along with adaptability to user-defined visual presentation preferences, is critical for providing an inclusive and adaptable user experience, particularly for individuals with low vision. This manual test empowers teams to diligently evaluate how content responds to user-driven resizing, spacing adjustments, and color overrides, moving beyond automated checks to catch nuanced issues that directly impact readability and usability. By meticulously addressing these concerns, platforms can ensure their content remains fully accessible and functional regardless of a user's visual needs or preferences.

User Location and Orientation
AAA
Purpose
The primary purpose of this manual accessibility test is to meticulously evaluate the accessibility of modal dialogs (also known as pop-ups or lightboxes) within a digital platform. This test is designed to be performed after an automated accessibility scan and aims to uncover complex issues related to focus management, semantic roles, and programmatic relationships that automated tools often miss. Ensuring that modal dialogs properly trap keyboard focus, correctly announce their role and purpose to assistive technologies, and do not obscure critical content is paramount. Without proper implementation, modal dialogs can create severe keyboard traps, disorient screen reader users, and render the underlying page content inaccessible, leading to a frustrating or impossible user experience..
WCAG Criteria Covered

This test specifically addresses the following WCAG 2.2 Success Criteria related to modal dialogs:

  • 2.1.1 Keyboard A

    This foundational criterion requires that all functionality of the content is operable through a keyboard interface. For modal dialogs, this means that the dialog itself must be launchable via keyboard, all interactive elements within the modal must be keyboard operable, and the modal must be dismissible via keyboard.

    • Relevance/Testing:

      We will verify that the trigger element (e.g., a button or link) that opens the modal is keyboard focusable and activatable. Once the modal is open, we will ensure that all elements inside it (buttons, form fields, links) are keyboard navigable and operable.
